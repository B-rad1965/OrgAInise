# OG DEV REFINEMENT: Revision Detection Impact Scan

## Goal

Improve the “Revise Memories” feature so it better detects affected memories when a user submits a major project revision.

This should be a refinement, not a rebuild.

Current behavior is generally good, but revision detection can miss affected memories even when focused search later reveals relevant conflicts.

## Observed Issue

When given a major Vealoria revision replacing the old “cosmic threat / ancient entity” direction with the new Axiom / Axiom Matter direction, OG correctly summarized the revision but proposed no memory changes.

However, manual focused search for “cosmic threats” later revealed several saved memories that were clearly relevant and should have been flagged for review, including:

* Shards tied to cosmic threats.
* Willow’s Veil entity connected to the larger cosmic threat.
* Open questions about the entity.
* Writing guidance referencing cosmic threats and entities.

## Desired Behavior

Before concluding “no changes proposed,” the revision system should perform an internal impact scan.

It should identify memories that may be affected by the revision, even if it is not confident enough to rewrite them.

The system does not need to automatically edit everything.

It should at minimum flag relevant memories for review.

## Suggested Process

When a revision is submitted:

1. Extract key changed concepts from the revision.
   Examples:

   * old concept being retired
   * new concept being introduced
   * affected entities
   * affected lore categories
   * affected characters
   * affected themes

2. Generate internal search terms from both retired and new concepts.
   Example from Vealoria:

   * cosmic threat
   * ancient evil
   * world eater
   * entity
   * Veil entity
   * Astralim
   * shards
   * Yuna origins
   * Axiom
   * Axiom Matter

3. Search saved memories using those terms.

4. Classify matching memories into safe action categories:

   ### Keep

   Memory is relevant but not contradicted.

   ### Review

   Memory may be affected but cannot be safely rewritten.

   ### Rewrite Suggested

   Memory clearly conflicts with the revision and a safe replacement is obvious.

   ### Archive Suggested

   Memory depends on a retired concept and should no longer remain active.

5. Show these proposed classifications to the user.

## Important Safety Rule

When uncertain, prefer “Review” over rewriting.

Do not aggressively overwrite memories based on inference alone.

## Example Expected Output

For the Vealoria revision, OG should have flagged:

* “The entity that notices Willow through the Veil is connected to the larger cosmic threat surrounding Yuna’s people.”

  * Suggested action: Review
  * Reason: The larger cosmic threat model has changed from an entity/world-eater concept to Axiom/Axiom Matter.

* “The shards are a key element in the cosmic threats facing Vealoria.”

  * Suggested action: Rewrite Suggested
  * Reason: The shards remain important, but their explanation should now connect to Axiom Matter rather than a traditional cosmic threat.

* “What is the nature of the entity that notices Willow through the Veil?”

  * Suggested action: Review
  * Reason: The existence and nature of the entity are no longer settled under the new direction.

## What Not To Change

Do not change the core memory storage model.

Do not change focused search.

Do not make revision edits automatic without approval.

Do not rewrite unrelated character memories unless the revision directly affects them.

## Success Criteria

After this refinement, major revisions should produce at least a short list of potentially affected memories when relevant matches exist.

The user should no longer need to manually search for obvious related conflicts after a revision.

The system should remain conservative, safe, and user-approved.
